# Архитектура федерации Tandi

**Статус:** черновик v1
**Дата:** 2026-04-21
**Цель документа:** карта, кто и как взаимодействует с федеративной инфраструктурой Tandi — чтобы до запуска Angular-пилота договориться о ролях, доступах и потоке изменений.

Этот документ — про **как**. Мотив и ценности — в `vision.md`, неизменные правила — в `constitution.md`, формат данных — в `schema.md`, онбординг координатора курса — в `curator.md`.

---

## 1. Актёры

В системе шесть ролей. У каждой — свой уровень доверия и свой режим общения с инфраструктурой.

| Роль | Кто | Чем управляет | LLM-ключ |
|---|---|---|---|
| **Core (доверенные агенты)** | Воробей + 2–3 человека, назначенных явно | ``, main repo, деплой флагманского курса | школьный/центрального узла |
| **Координатор курса** | волонтёр, ведёт один курс | `courses/<название>/config/`, свои промпты, свой релиз | свой |
| **Activist** | студент/выпускник с собственной подпиской Claude/Gemini/Codex | свои агенты, читающие публичные данные; предлагает PR-ы | свой |
| **Student (baseline)** | студент текущего потока | свой репозиторий-дневник | обычно нет |
| **Mentor / juror** | внешний инженер, проверяет защиты | свой агент (опционально) для подготовки вопросов | свой |
| **Public reader** | кто угодно, читает дашборд | ничего | нет |

**Ключевая идея:** core не кормит агентов активистов и менторов своими ключами. Каждый приходит со своим LLM, инфраструктура лишь выставляет **контракты** и **данные**.

## 2. Схема взаимодействий

```
                       ┌──────────────────────────────┐
                       │  CORE / SCHOOL               │
                       │  (trusted agents)            │
                       │                              │
                       │            │
                       │  ├── constitution.md   ◀──┐  │
                       │  ├── schema.md            │  │
                       │  ├── curator.md           │  │
                       │  └── wiki/                │  │
                       │                           │  │
                       │  RS Tandem (флагман)         │  │
                       └──┬──────────────┬─────────┼──┘
                          │ RFC / PR     │ read    │
                          │              │         │ RFC / PR
           ┌──────────────┴──┐     ┌─────┴────────┴─────┐
           │ КООРДИНАТОР     │     │ ACTIVIST           │
           │ КУРСА           │     │ (own subscription) │
           │ (Angular, ...)  │     │                    │
           │                 │     │                    │
           │ courses/X/      │     │ свой агент         │
           │ ├── config/     │     │ читает публичные   │
           │ ├── prompts/    │     │ данные через MCP   │
           │ └── data/       │     │ или HTTP           │
           └──┬──────────────┘     └────┬───────────────┘
              │ publishes              │ opens PR / issue
              │ weekly issue           │ for data fix
              ▼                        ▼
     ┌────────────────────────────────────────────┐
     │  STUDENTS / MENTORS / PUBLIC READERS       │
     │  - читают дашборд                          │
     │  - ведут дневники в своих репо             │
     │  - приходят на защиты                      │
     └────────────────────────────────────────────┘
```

Стрелки — направление управления, не направление данных. Core задаёт контракты, все остальные их читают; обратное движение (правки, amendment-ы) идёт через PR/RFC, а не через прямую запись.

## 3. Юз-кейсы

Семь сценариев, от повседневного до редкого. Каждый — проверка: выдерживает ли архитектура?

### UC1. Координатор заводит новый курс

**Ситуация:** Маша хочет запустить курс по Python весной 2026.

**Поток:**
1. Маша читает `vision.md`, `constitution.md`, `curator.md` (≈45 минут).
2. Форкает репу или работает в форке монорепо.
3. Создаёт `courses/python/` по шаблону Angular: README, `config/participants.json`, `config/program-config.json`.
4. Локально прогоняет первый `collect → scan → write` на одной команде своим LLM-ключом.
5. Проверяет голос Tandi по чек-листу из `constitution.md` раздел 6.
6. Открывает PR в школьный монорепо: «новый курс python».
7. Core-агент ревьюит: совпадает ли форма данных со схемой, не ломает ли общий `tandi-index`, не тянет ли новых зависимостей в core.
8. Мердж → курс стартует.

**Что может сломаться:** Маша добавила поле в participants.json, которого нет в schema. Ревью ловит. Предлагает: либо подать amendment к schema (RFC), либо держать поле внутри своего курса (`courses/python/config/extras.json`).

### UC2. Активист спрашивает свой прогресс

**Ситуация:** Антон учится на RS Tandem, у него Claude Pro. Хочет, чтобы его агент сам следил за его же прогрессом.

**Поток:**
1. Антон подключает своего агента к публичным данным школы (см. раздел 5 про транспорт).
2. Агент читает: его текущий балл, сколько дневников осталось, что он писал в W3, какие вопросы ему выдали к защите.
3. Агент **не пишет** за Антона дневник. Роль агента — тренер, не замена.
4. Антон пишет сам, может попросить агента отзеркалить: «это запись уровня 2 или 3 по шкале из `angular-intro.html`?».

**Что может сломаться:** Антон просит агента «напиши мне дневник за неделю». Тут ответ — политика, не архитектура: конституция запрещает подменять авторство. Агент должен отказать, и это в промпте personal-agent template, который мы опубликуем. Но проверить факт замены — невозможно без человеческого ревью при защите. Живая защита и есть этот фильтр.

### UC3. Активист чинит данные

**Ситуация:** Антон видит: его команда на дашборде без одного участника. Очевидная ошибка в `participants.json`.

**Поток:**
1. Его агент (или он сам) открывает issue в `rollingscopes/rs-tandem-observer` по шаблону `data-correction`.
2. Trusted-агент или сам Воробей проверяет, чинит `participants.json`, прогоняет `collect`, деплой.
3. Комментарий в issue, закрытие.

**Что может сломаться:** поток залит мусорными issues от агентов-энтузиастов. Защита — issue-шаблоны с обязательными полями и rate-limit по логину.

### UC4. Координатор предлагает поправку к конституции

**Ситуация:** координатор английского курса обнаруживает: правило «писать Tandi по-русски с тёплой интонацией» не переносится в английский курс без потери голоса.

**Поток:**
1. Координатор открывает RFC-issue: `[RFC] amendment: voice rules for non-Russian courses`.
2. В теле — описание проблемы, конкретный пример, предлагаемый текст изменения.
3. Обсуждение в issue. Core-агенты комментируют.
4. Если договорились — PR с правкой `constitution.md` + запись в `wiki/log.md` с датой, автором, мотивом.
5. Мердж требует подтверждения от ≥2 core-агентов (branch protection + CODEOWNERS).
6. Остальные курсы подтягивают обновлённую конституцию при следующем `git pull`.

**Что может сломаться:** конституция разрастается в юридический документ. Защита — каждый amendment должен либо заменять существующее правило, либо иметь явное обоснование, почему нужно новое.

### UC5. Ментор использует своего агента для подготовки защиты

**Ситуация:** Ментор Сергей защищает пять студентов в субботу. У него есть Claude Pro.

**Поток:**
1. Сергей скачивает лист вопросов своих студентов (публичная ссылка на `mentor-sheets/`).
2. Его агент читает связанные дневники через MCP-фид школы.
3. Агент генерирует follow-up вопросы на лету во время защиты.
4. Школа не знает о существовании этого агента. Это приватный инструмент ментора.

**Что может сломаться:** агент ментора галлюцинирует факт про студента и ментор его озвучивает. Защита — не архитектурная, а процессная: ментор отвечает за то, что задаёт. Агент — помощник, не авторитет.

### UC6. Cross-course digest

**Ситуация:** у нас три активных курса (Tandem, Angular, Python). Хочется еженедельный «общая картина».

**Вопрос:** кто пишет? Core-Tandi? Отдельный агент? Ротация между координаторами?

**Предварительный ответ:** одна из core-обязанностей — еженедельный `garden-digest`, который собирает последние issue каждого курса и пишет короткое «что произошло за неделю». Запускается из core-репо, читает `courses/*/data/issues/latest.json`. Это — один из редких случаев, когда core пишет поверх всех курсов, и это должно быть read-only: digest не меняет данные курсов.

**Что не решено:** публикуется ли digest внутри core-дашборда или в отдельном канале. Пока — внутри, как новая вкладка `#/garden`.

### UC7. Брошенный курс

**Ситуация:** координатор запустил курс, через 3 недели пропал. Курс стоит.

**Поток:**
1. Студенты сигналят в core-issue.
2. Core-агенты переключают курс в **архивный режим:** данные остаются читаемыми, сбор останавливается, баннер на странице «курс заморожен с даты X, причина».
3. Если кто-то хочет подхватить — это новый онбординг координатора на существующие данные.

**Что может сломаться:** доступ к репозиториям студентов останется у ушедшего координатора (его токен). Защита — токены живут в курсовом конфиге как **ссылка на secret store**, не как значение; при передаче курса токен ротируется.

## 4. Кольца доступа

Четыре кольца, каждое включает всё, что внутри.

```
  Ring 0 ── core / trusted agents
           └── write: *, merge to main
  Ring 1 ── координатор курса
           └── write: courses/<name>/*
  Ring 2 ── student / activist
           └── write: свой дневник в своём репо; PR/issue в школьный репо
  Ring 3 ── public reader
           └── read: публичный дашборд, публичные MCP-фиды
```

**Как это enforced:**

- **Ring 0 → Ring 1:** CODEOWNERS. `**` и `main` бранч защищены, требуют ревью от core.
- **Ring 1 → Ring 2:** каждый координатор — CODEOWNER на свой `courses/<name>/**`. Без этого любой PR в чужой курс требует согласия core, а это лишнее трение.
- **Ring 2 → Ring 3:** PR-режим. Активист не получает прямой write; его изменение проходит ревью владельца того курса, куда он пишет.
- **Публичные данные:** всё, что в `public/` и в `` — публично. Секреты (токены, LLM-ключи) — только в secret store провайдера (GH Actions secrets или локально у координатора).

**Что не надо делать:** специальных «junior trusted» уровней, квот, кармы. Всё это — оверхед на пустом месте при текущих объёмах. Начинаем с трёх ring-ов + CODEOWNERS, усложняем только если реально сломалось.

## 5. Транспорт: как агенты читают/пишут данные

Три способа, по убыванию официальности.

### 5.1. Git — основной канал

Правки данных, конфигов, документов — через PR. Это дорого по трению, но даёт аудит, ревью и откат. Для **любых записей** это должен быть дефолт.

### 5.2. HTTP — для чтения публичных данных

Публичный дашборд уже раздаёт JSON (`public/data/`). Личный агент студента может просто фетчить `https://rollingscopes.github.io/rs-tandem-observer/data/config/participants.json` и жить. Никакой авторизации не нужно — это публично.

Минус: форма JSON может меняться, внешние агенты ломаются. Защита — `schema.md` как контракт + `tandi-index.json` как версионированная точка входа.

### 5.3. MCP — для структурированного доступа

Три MCP-сервера, которые core может выставить **в будущем**, если найдётся ресурс на поддержку:

- `tandi-data` (read-only) — участники, чекпоинты, выпуски, баллы
- `tandi-garden` (read-only) — текущий конституция, schema, wiki
- `tandi-voice` (tools) — один инструмент `check_voice(text) → report` для самопроверки голоса; используется и координаторами, и активистами

**MCP не заменяет git для записи.** Запись — всегда через PR. MCP — удобный слой чтения, который делает пересадку между LLM дешевле (сменить Claude на Gemini внутри курса = поменять клиента, не переписать интеграции).

**Приоритет:** сначала schema + git-поток. MCP — когда будет ≥2 независимых активиста, которые реально хотят подключить агентов. Не строим впрок.

## 6. Поток изменений

Четыре класса изменений, четыре процедуры.

### 6.1. Фикс данных (fast path)

Шаблон: `data-correction`. Примеры: неправильное имя в команде, битая ссылка на репо.

- Issue → core или координатор курса чинит → re-collect → deploy → закрытие issue.
- SLA: в разумное время; без церемоний.

### 6.2. Изменение внутри курса (course-scoped)

Примеры: новый чекпоинт, новая рубрика, поправка промпта.

- PR от координатора в свой `courses/<name>/`.
- Ревью: автоматика (типы, тесты, схема) + core-агент смотрит «не ломает ли shared schema».
- Мердж — координатор.

### 6.3. Конституция / schema / core-процесс (RFC)

Примеры: новое правило голоса, новое поле в schema, смена принципа доступа.

- RFC-issue с шаблоном: проблема, предложение, альтернативы, прецеденты.
- Обсуждение публично.
- PR с правкой + запись в `wiki/log.md`.
- Мердж: ≥2 core-агента (enforced через branch protection).

### 6.4. Incident / voice drift

Примеры: Tandi выдала что-то, что противоречит конституции. Кто-то публично заметил.

- Issue с шаблоном `incident-report`: ссылка на публикацию, раздел конституции, который нарушен, предложение починить.
- Core-агент: признать → исправить на месте (если возможно) → записать в `wiki/incidents/<slug>.md`.
- Если паттерн повторяется — это триггер для RFC на уровень конституции.

## 7. Модель доверия

**Как становятся core-агентом.** Приглашение от Воробья. Явно, письменно, в `wiki/log.md`. Выход — такой же явный.

**Как становятся координатором курса.** Заявка через issue `course-request`. Core смотрит: есть ли у человека контекст RS School (был студентом, ментором, автором курса), есть ли опыт вести группу, согласен ли с конституцией. Приглашение — подтверждённое письменно.

**Как становятся активистом.** Никак — это не роль, это режим. Любой с подпиской может открыть PR или натравить своего агента читать публичные данные.

**Как теряют доверие.**
- **Core:** массивное нарушение конституции с умыслом. Редкий случай, но процедура должна быть.
- **Plot curator:** бросил курс (UC7). Не «уволен», курс просто переводится в архив; координатор остаётся в списке, но без активных прав.
- **Активист:** токсичное поведение в issue/PR → ban на уровне GitHub, процедура как у любого open-source.

**Что мы сознательно не делаем:** сложные оценки «кармы» агентов, автоматическое повышение, delegated trust. Это преждевременно при любом размере меньше ~50 активных участников.

## 8. Публичный vs приватный слой

**Публично (анонимно читают через HTTPS или MCP):**
- Списки команд и участников (GitHub-логины, имена — они и так публичны)
- Еженедельные выпуски Tandi
- Баллы по чекпоинтам
- Схема, конституция, wiki
- Листы вопросов к защитам

**Приватно (в secret store):**
- LLM-ключи (каждого координатора отдельно)
- GitHub токены (для Octokit)
- Персональные заметки ментора про студента (если такие появятся — но пока не появились)

**Не персональные данные, которые тем не менее стоит думать:**
- Оценки жюри по студенту — публикуем агрегированно (средняя), не по жюри (решение зафиксировано ранее по работе с финалами)
- Негативные сюжеты про конкретного студента в hero/fail — только с явным согласием; это правило голоса Tandi, не архитектуры

## 9. Rejected use cases и что ещё не решено

### Rejected

- **UC6 — cross-course digest.** Рассматривался как еженедельный сводный пост «что произошло в федерации за неделю», собираемый core-репо поверх `courses/*/data/issues/latest.json`. **Отклонено в ходе multi-course-layout (2026-04-22).** Причины: (1) на момент отклонения активен только один курс + пилот Angular — нет двух курсов, которые реально надо сравнивать; (2) cross-course post — это отдельный голос/роль, которую мы ещё не определили; (3) трюк «пишем поверх всех курсов» ломает принцип «курс — автономная единица». Возвращаемся к вопросу не раньше, чем у трёх курсов параллельно будет >=2 выпусков.

### Нерешённое

Нерешённое держим здесь, не в головах. Когда решается — переносим в соответствующий раздел и удаляем отсюда.

- **Монорепо vs separate repos.** Сейчас монорепо (`courses/` внутри `rs-tandem-dashboard`). Порог, после которого это становится больно: ~3 активных курсов с независимыми релизами. До тех пор — монорепо.
- **Issue-репозиторий.** Используем ли `rollingscopes/rs-tandem-observer` для всей федерации, или заведём отдельный `rs-tandem-garden`? Предварительный ответ — пока один, разделим когда забьётся.
- **RFC-шаблон.** Черновик есть в этом документе; выписать отдельно в `.github/ISSUE_TEMPLATE/` когда будет первый реальный RFC.
- **MCP-сервера.** См. 5.3. Не раньше второго активного участника.
- **Voice-drift детекция.** Автоматика или человеческое ревью? Пока — человеческое (раз в неделю core перечитывает свежие публикации). Автоматика — если доживём до 5+ курсов.
- **Синхронизация конституции в форках.** Если координатор работает в отдельной репе, как быстро он подтягивает обновления? Пока — рекомендация в `curator.md` «раз в неделю `git pull upstream main`». Если не работает — думаем про submodule или sync-скрипт.
- **Стоимость LLM.** Каждый координатор платит за свой ключ из своего кармана. Если появится спонсор пула — это отдельное решение, не архитектурное.

---

## 10. Мета

Этот документ — **карта, не контракт**. Контракты живут в `constitution.md` и `schema.md`. Карту можно править без RFC — достаточно PR и здравого смысла. Но если правка меняет одно из кольец или процедур, она тянет RFC на уровень конституции.
